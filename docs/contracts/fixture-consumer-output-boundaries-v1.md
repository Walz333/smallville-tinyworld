# Fixture Consumer Output Boundaries v1

## Purpose

This document defines the non-negotiable boundaries for diagnostic outputs derived from canonical fixtures by approved consumer profiles.

## Core Boundary Rules

- Outputs are derived only.
- Outputs are not canonical fixtures.
- Outputs may not replace canonical source text.
- Outputs may not weaken uncertainty.
- Outputs may not promote inferred content to observed.
- Outputs may not imply runtime state.
- Outputs may not imply execution approval.
- Outputs may not imply Water Mill real-world authority.
- Outputs may not introduce a second dossier.
- Outputs may not soften earlier stop-rule or conformance boundaries.

No diagnostic output family may create a new authority layer above the canonical fixture text.

## Canonical Source-of-Truth Boundary

Canonical fixture files at canonical paths remain the source of truth. Derived outputs may describe, compare, summarize, group, or report boundary conditions, but they may not replace, rewrite, mirror, supersede, or silently normalize the canonical source text.

## Water Mill Boundary

Diagnostic outputs must preserve that Water Mill content remains design-asset-derived context only. No downstream wording may restate Water Mill design assets as observed built reality without accepted evidence.

## Packet Boundary

Diagnostic outputs must preserve that the promoted derived packet remains an interpretive artifact only and not runtime world state. No output may present packet language as world-state authority, runtime instruction, or scene mutation authority.

## Single-Dossier Boundary

Diagnostic outputs must preserve single-dossier continuity. No output may silently merge, imply, introduce, or normalize a second dossier.

## Non-Authoritative Output Boundary

All diagnostic outputs remain non-authoritative and non-replacing. Review-oriented language is permitted. Authority-creating language is not.

## Review-Only Guidance Boundary

`recommended_action` is review-only follow-up guidance and may not authorize mutation, runtime action, execution, or canonical replacement.

## No-Implementation Boundary

This layer does not define tooling behavior, validator behavior, schema behavior, packaging, file formats, runtime coupling, access control, scripts, or implementation flow.

## Explicit Stop Conditions

Stop if any output family or output wording:

- implies runtime authority
- implies mutation authority
- implies execution authority
- implies canonical replacement
- softens a triggered stop condition into advisory-only wording
- reframes packet content as runtime world state
- inflates Water Mill design-asset-derived context into observed built authority without accepted evidence
- weakens single-dossier continuity
- drifts into tooling, validators, scripts, mirrors, schemas, packaging, or runtime-coupled behavior
