# UEIA Shadow Trial Runbook v1

## Purpose

This runbook defines the accepted human-reviewed flow for UEIA Autonomous Ops Trial Phase 1.

It is operational guidance for shadow-mode only. It does not authorize implementation, runtime launch, or runtime mutation.

## Baseline And Operating Rules

- Phase 1 status: `accepted` for documentation and schema drafting only
- Host-native default: `accepted`
- Smallville runtime containerization: `deferred`
- Packet status: reviewable interpretive artifact only
- Dossier rule: one constrained dossier only

This runbook must not be used to justify:
- live runtime mutation
- automatic proposal approval
- connector-driven autonomy
- container-required workflow

## Human-Reviewed Flow

### 1. Choose Source Dossier

Choose one constrained dossier only.

Do not merge multiple dossiers or synthesize across dossiers unless separately approved.

Bootstrap recommendation only:
- accepted post-fix `two-house-garden-v1` reviewed run
- accepted post-fix `tiny-world` smoke as control context

### 2. Ingest Approved Evidence

Ingest only approved snapshot sources:
- accepted run bundles
- approved local notes
- accepted archive material already in scope
- approved static design assets as snapshot references only

Do not ingest:
- live connectors
- scraping results
- runtime hooks
- unreviewed feeds

### 3. Normalize To Archive Record(s)

Normalize the dossier into archive-record drafts with explicit fact status.

Expected analytic work may include:
- chronology reconciliation
- structure classification
- map-to-space alignment
- flow interpretation
- hypothesis logging
- archive cross-reference
- thesis question extraction

No derived fact may appear without `fact_status`.

### 4. Compile Derived Scenario Packet

Compile a reviewable interpretive packet from the draft records.

The packet may describe:
- spatial context
- built features
- water-system elements
- visualization hints
- constraints
- open questions

The packet must not be treated as:
- runtime world state
- direct scene mutation authority
- direct world mutation authority
- geometry or scene asset specification

### 5. Compile Agent Brief(s)

Compile reviewable brief drafts from the packet and record drafts.

Any background or low-priority analytic tasks remain proposal-only intellectual work.

They do not authorize execution, mutation, or autonomous expansion.

### 6. Produce Proposal And Review Pack

Assemble the packet, briefs, open questions, provenance notes, and review findings into a proposal and review pack.

Apply the materials / people / power / place lens:
- Materials: what is directly evidenced versus inferred?
- People: whose labor, maintenance, or access is implied?
- Power: what unsupported control assumptions are being introduced?
- Place: what spatial claims are well evidenced, and what remains interpretive?

### 7. Stop For Human Approval

Stop at the human approval boundary.

No automatic approval path is allowed in this runbook.

## Structure Evidence Ladder

Keep these categories distinct during review:
- directly evidenced structures
- inferred structures
- reconstructed or designed structures

No designed or reconstructed structure may be treated as `observed` unless anchored by accepted source evidence.

## Design-Asset Snapshot Rule

Approved static design assets are snapshot and reference inputs only.

They do not grant:
- observed real-world status
- runtime geometry authority
- world-state authority
- implementation authority
